Part Overview

Main Control Logic (loop)

The main decision-making logic of the system is located at the end of the loop() function. This section coordinates feeding, monitoring, and user interaction through a series of conditional checks.

  • if (settingState == NOT_SETTING && manualState == MANUAL_IDLE)
    Ensures that automatic feeding only occurs when the user is not editing settings and is not performing a manual feed.
  • if (feedingActive)
    Continuously monitors the feeding process during every loop cycle.
  • if (millis() - lastScreenUpdate > 1000)
    Updates the LCD once per second to prevent flickering and unnecessary performance overhead.

Scheduled Feeding Check

This function determines whether a scheduled feeding should be triggered. It ensures accurate timing, prevents duplicate activations, and validates feeding conditions.

  • Checks whether the feeder is already active.
  • Obtains current time from the RTC (or simulates time using millis()).
  • Iterates through all feeding slots.
  • Validates slot enable status and assigned food weight.
  • Matches scheduled hour and minute exactly.
  • Prevents multiple triggers within the same minute.
  • Stores last trigger time as a memory checkpoint.
  • Initiates feeding when all conditions are met.

startFeeding()

This function initializes a feeding session but does not directly dispense food. Instead, it prepares all states and safety mechanisms required for controlled feeding.

  • Identifies the feeding type (scheduled or manual).
  • Sets feeding-related state flags.
  • Computes target weight and required food amount.
  • Initializes timing and jam-detection safety tracking.
  • Outputs debug information via serial monitor.
  • Activates the feeder mechanism.
  • Updates the user interface.

Manual Feeding Initialization

Manual feeding follows a similar initialization process as scheduled feeding, but is triggered directly by user input and does not rely on predefined feeding slots.

Feeding Monitor (Scheduled & Manual)

This function continuously monitors the feeding process and ensures safe and accurate operation.

  • Runs only when feeding is active (guard clause).
  • Caches key sensor values to reduce noise and CPU load.
  • Stops feeding once the target weight is reached.
  • Detects feeder jams or lack of food flow.
  • Enforces hard safety timeouts.
  • Logs periodic feeding status.

finishFeeding()

This function finalizes the feeding session, whether completed normally or terminated due to an error.

  • Captures the feeding context before reset.
  • Logs feeding results and events.
  • Resets all feeding-related states.
  • Validates system state and provides debug feedback.
  • Updates the LCD with user feedback.

resetSystemState()

This function performs a clean system reset, ensuring the feeder returns to a known safe idle state.

  • Resets all feeding schedules and slot configurations.
  • Clears feeding history records.
  • Reinitializes weight sensors.
  • Forces hardware and UI into a safe closed state.

Setting Mode Handling

This logic manages scheduled feeding configuration, following the flow: enter mode → set hour → set minute → set weight → save → exit.

Adjust Setting Value

Handles value incrementation and decrementation within the setting mode, directly supporting user interaction.

Save Current Slot

Stores edited schedule parameters and provides immediate visual feedback on the LCD.

Add Feed Log

Records completed feeding events into the system history for tracking and analysis.

  • Shifts older logs to maintain buffer space.
  • Retrieves current time from RTC or simulated clock.
  • Stores feeding metadata, target weight, and final weight.
  • Safely manages log count to prevent overflow.
